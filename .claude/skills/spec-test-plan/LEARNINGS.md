# LEARNINGS — spec-test-plan

Accumulated procedural wisdom across projects. Read at task start; append at task end. Prune and promote quarterly.

Keep entries specific (cite bug class + evidence). Vague entries get pruned. Soft cap ~100 lines — promote recurring ones into SKILL.md and archive raw entries.

## What Worked

- **[handler run-parameters coverage verification]** — Codex Std/Adv caught gaps that the first draft matrix made look covered: serverless async invoke has a whole-payload size limit distinct from the handler START budget; legacy event-bus targets can break during rolling deploy before reconciliation rewrites them; new DB columns need both metadata and API/cache gates; scheduled jobs need explicit stage plus stage-scoped IAM; string/json NUL and malformed Unicode can fail persistence after resolver success; DST date/datetime strings must survive UI/API/event-bus without `Date` coercion. Pattern: for handler-parameter specs, add composed payload-size tests, legacy-target deploy gates, schema-cache deploy probes, stage/IAM infra-as-code assertions, persistence-encoding tests, and schedule review→resync chain tests before coverage verification closes.
- **[profile setup/orders coverage verification]** — Codex coverage reviewers caught product-reality gaps after the first test-plan draft: private dashboard saved-card/order flows must stay unchanged; setup readiness must be store/account-scoped across two users, not session-scoped; full approved-retailer setup-to-order journey is mandatory; rollback flag drills, money precision, and cloud-runtime service-account secret/payment-processor probes belong in the deploy tier. Pattern: for dashboard-first order-gating specs, add private-scope unchanged rows, full setup-to-order journey E2E, rollback drill, Decimal-to-integer-cent contract, and deployed service-account payment probe before considering coverage complete.
- **[proof-of-delivery / PII coverage verification]** — Codex coverage reviewers found gaps after a strong first-draft plan: heavy PII fields can be omitted from serializers but still loaded by default querysets/exports; idempotency needs tested unique constraints, TTL cleanup behavior, and timeout replay after cleanup; mobile-submodule evidence must be machine-checkable before flow promotion. Pattern: for PII-bearing mobile/backend specs, add explicit rows for DB projection/export denylist, idempotency persistence/cleanup, and release-gate artifact checks before running coverage verification.
- **[backend + mobile-submodule surface extraction]** — When a spec spans a backend repo plus a mobile submodule, the surface extractor must verify whether the mobile tree is populated before assigning mobile test rows. An empty mobile-app worktree turns mobile rows into explicit blocking prerequisites / separate-PR requirements, while backend serializer/API contract rows can still proceed in the main repo.
- **[multi-lane V1 final verification]** — Aggregated final verification needs its own flow-backed plan shape, not just N lane plans plus chat memory. The useful artifact was a command matrix (`testing/v1-final-verification.json`) that composes lane ledgers, backend suite commands, staging browser batches, and mobile/iOS smoke. Pattern: when 3+ feature lanes converge, add an aggregate "final verification" flow/test matrix that references the lane ledgers and makes non-PASS ownership explicit before spec-test-execute starts.
- **[customer-payments-v1 test plan]** — Codex Standard caught a count-mismatch self-contradiction: a traceability matrix referenced a typo'd test ID AND a row range (8 rows) while the spec had 11 customer-state labels. Both invisible to the agent that wrote them. Pattern: when a coverage matrix uses ID ranges instead of explicit IDs, Codex's count-vs-spec-text comparison exposes silent under-coverage. Always make the matrix list named tests, not "T2-X..T2-Y" ranges, OR ensure both ends are explicit.
- **[customer-payments-v1]** — Codex Adversarial caught a deploy-tier failure mode the 3 analysis agents missed: a scanner endpoint may be unauthenticated (publicly callable) OR the scheduler service account may lack invoker permission — either way, the scanner never runs in prod despite passing local tests. Pattern: any spec that adds a scheduled job MUST have a deploy-validation row that probes the *actual scheduled invocation* with the production IAM principal, not just an HTTP smoke test.
- **[customer-payments-v1]** — Codex Adversarial (webhook endpoint subscription set) — the test plan asserted the V1 webhook URL was registered, but did NOT assert the subscribed event SET matches the V1 event matrix (15 events). Production failure mode: register endpoint with happy-path 5 events, then the other events fire and silently drop. Pattern: every webhook integration spec needs a deploy-validation row that asserts subscription set equals spec set, not just URL parity.
- **[customer-payments-v1]** — Pre-existing real bug elevated by the Adversarial Analyzer: two cancel-order paths saved `status='cancelled'` and `order.save()` WITHOUT releasing the payment authorization. This leaves an authorized PaymentIntent on the payment processor (auth hold for days, customer sees a pending charge). Surfaced via direct file Read of cancel-view source — the agent verified the actual code, not the spec's claim. Pattern: when a spec claims X happens via signal/handler, instruct agents to verify by reading the actual signal/view, not the spec's prose.
- **[customer-payments-v1]** — Codex coverage caught an over-tested CSP cluster: the same CSP directives were tested 3 ways across integration and deploy tiers. Pattern: CSP coverage should split as middleware-header-construction (unit) + prod-proxy-CSP (deploy) + browser-rendering-under-CSP (E2E), with NO redundant integration tier. Removed the redundant integration rows; kept the deploy + browser path.

- **[handler-async-dispatch test plan]** — Codex Adversarial coverage caught 3 CRITICAL prod-only gaps that none of the 3 analysis agents (Infra Detector + Surface Extractor Opus + Adversarial Analyzer Opus) surfaced: (1) real-environment JWT scope mismatch from a CLI test JWT (every async run could reap as no_heartbeat in prod despite passing staging), (2) prod runtime-image freshness pre-flip gate (flipping the flag dispatches into a stale runtime), (3) queue dispatch-level idempotency under executor crash + redelivery (completion idempotency is per-token, dispatch-level is undefined → duplicate side effects). Pattern: the 3-agent dispatch reasons about the spec; Codex coverage reviewers reason about what would BREAK in prod given the test plan. Both passes are necessary; coverage verification is NOT optional.
- **[handler-async-dispatch]** — Two reviewers independently flagged the same conflict (E2E token capture from worker logs vs a token-redaction invariant) — high-confidence signal that justified solving once via a test-only proxy abstraction rather than a per-test workaround. Pattern to promote: when Codex Std AND Codex Adv flag the same surface, the resolution is usually a SHARED abstraction, not a per-test fix.
- **[handler-async-dispatch]** — A `process.exit(139)` "worker crash" E2E was a unit-only simulation flagged by Codex Adv (the process API is unavailable in the isolated-vm context). Switched the mechanism to an EXTERNAL SIGKILL via the sandbox's exec interface against the worker process. Pattern: when synthesizing crash tests for a sandboxed/isolated runtime, the kill must come from OUTSIDE the isolated context — a user-handler-side `process.exit` does not actually crash the supervisor in this architecture.
- **[handler-async-dispatch]** — A pre-Step-2 Infra Reality Audit identified that the worker module had ZERO tests, the stalled-run-checker had ZERO, and the large MCP-server test file covered no async-dispatch tools. The audit's "NEW test files required ~9 / ~2600-3300 LOC" estimate held up exactly through Steps 2-7. Pattern: the reality audit's effort estimate is load-bearing for users to plan implementation budget; specifically calling out which existing files need EXTEND vs NEW makes the test-plan's scope concrete from the start.
- **[HTTP-only DB-edge rewrite]** — Surface Extractor on Opus surfaced a cross-org cache isolation gap (a per-org invalidation row covered single-org but not cross-org bleed of in-memory client/schema caches and cache keys), `force_fresh` bypass-AND-skip-write-back semantics (must verify both halves), 17 cross-cutting invariants, and tenant-gap edge cases (orgs without a configured secret ARN). Sonnet would have under-extracted these. Confirms the LEARNINGS pattern: Opus on the surface extractor is load-bearing whenever the spec touches multi-tenant state or has cross-cutting platform invariants.
- **[HTTP-only DB-edge rewrite]** — Plumbing trace **dynamic introspection** (walk every handler registered with `readOnlyHint === false` at runtime) catches what a hardcoded list misses. The original plan enumerated 6 write tools; the actual count was 10 (4 write tools were missing from the list). Same shape as a prior callback-wiring regression where a mutator lacked its wired completion callback. Pattern to promote: never hardcode plumbing-trace tool lists — always introspect the registration surface and fail-loud when a new mutator lacks the wired callback.

- **[DB-edge sync overhaul]** — Cross-cutting / invariant rows referenced in the traceability matrix but missing from any tier table is a recurring blind spot. A coordinator wrote a matrix referencing `X-10..X-14` while never defining those rows in a tier table; Codex Std caught all 3 instances. Pattern: when surfacing "cross-cutting invariant" rows from the surface extractor's output, ALWAYS land them in their own Cross-Cutting tier table BEFORE referencing them in the matrix. Otherwise traceability becomes a lie.
- **[DB-edge spec]** — Codex Adversarial coverage caught two CRITICAL deploy-gate gaps that no Claude reviewer surfaced: (1) a Phase 0 spike result needs a MECHANICAL deploy-block (a JSON artifact + CI gate), not just spec text; (2) a shadow-compare alarm needs a MANDATORY runbook action (halt-cutover/rollback), not just "page". Pattern: "page on X" without an automated halt action is operationally weak — every divergence/critical alarm needs an explicit downstream action wired in the test plan. Add a new Adversarial Analyzer category for the next run: "every alarm has a paired automated action".
- **[operator-authoring spec]** — Codex coverage reviewers caught 4 spec bugs the test plan couldn't catch on its own, because they required re-reading the spec with tests as a lens: (1) `committed_at: null` in a NoSQL initial put is not equivalent to `attribute_not_exists` — NULL attributes exist; silent first-commit failure. (2) A `draft_hash` covered tree+filters_schema+kit_version but missed path/title/description/icon/sort_order metadata — preview-vs-commit metadata drift attack. (3) An object-store key with second-resolution (`YYYYMMDDTHHMMSSZ`) produced nondeterministic sort order on same-second writes. (4) A NoSQL 400 KB item limit had no structured error path. Pattern: **always run Codex Std + Adv coverage reviewers after writing the test plan — they find spec semantic bugs tests alone can't surface**, because they compare plan assertions against spec claims with cold eyes.
- **[operator-authoring spec]** — A cross-tenant orgSlug matrix for every orgSlug-taking tool was missing from the first-draft plan, even though the spec had a single-line "cross-org 403" requirement. Codex Adv caught this as CRITICAL: a per-tool matrix asserting same-org succeeds / wrong-org 403 / zero side effects. Generalize: when a spec has an aggregate rule ("X is org-scoped"), the plan must enumerate the rule's application per tool, not assert the aggregate once.
- **[operator-authoring spec]** — Rolling-deploy N-1/N chaos was missing from the first-draft plan despite an explicit rolling-deploy mention in the spec. Codex Adv called this CRITICAL. Pattern: any spec with `template_version` / `tool_contract_version` drift handling needs a real chaos test where the weighted alias has both versions live simultaneously, not just a unit test of the error code.
- **[handler-runtime]** — Feeding flow-registry gotchas as "BLOCKING operational knowledge" (NOT optional commentary) into the surface-extractor and adversarial-analyzer prompts. An async-invoke-hides-errors gotcha from an automation-trigger-run flow directly informed the result-marker-parse-failure + sync-invoke regression guards. Without the flow context, the marker parsing would have been a generic "test happy path" checkbox.
- **[handler-runtime]** — 3-agent parallel dispatch (Explore infra + Opus surfaces + Opus adversarial) producing ~200 surfaces + 79 adversarial items from a 968-line spec. Opus on surfaces is load-bearing: Sonnet cannot cross-match 113 surfaces against 16 adversarial categories while maintaining plumbing-trace rigor.
- **[handler-runtime]** — Codex coverage adversarial review caught three things the 3-agent dispatch missed: (1) CDN auth-vs-cache (a cache key excluding the bearer = cross-tenant leak), (2) direct global-fetch egress bypassing sealed transport (the ADV categories didn't explicitly cover "handler bypasses ctx.tools entirely"), (3) prod-vs-staging DB migration drift at the deploy gate (not just staging). All three classes should join the adversarial-analyzer prompt pattern for future runs.
- **[blast-radius categories]** — Adversarial-analyzer categories 10–16 (concurrency, state-ordering, plumbing-trace, tenant variance, observability, runtime parity, consumer contracts) surfaced tests the generic "env divergence / format mismatch" categories missed. Real-bug evidence in prompts keeps the categories grounded, not abstract.
- **[consumers column on every mutation]** — Forcing the surface-extractor to enumerate every callback / serializer / admin-view / mobile-decoder / downstream-job reader per mutation caught plumbing drops before they shipped (retroactively would have caught a real integration helper wired into only some of the completion paths).
- **[chain tier 2.25]** — Adding an explicit tier between integration and E2E for multi-step sequences (upload→serialize→render; checkout→execute→checkin) catches plumbing bugs unit+integration miss without full-browser overhead.
- **[treat flow gotchas as blocking operational knowledge]** — Not optional commentary. Frame them to analysis agents as "protocol instructions that prevent test failure." Prevents rediscovery across cycles.

## What Failed

- **[HTTP-only DB-edge rewrite]** — Codex Std + Adv coverage reviewers BOTH hallucinated against the OLD test plan content despite reading the rewritten spec. They cited test row IDs that didn't exist in the current 269-line plan — all from a prior 707-line sidecar plan. Cause: the spec's Background and Rewrite-history sections mentioned old-architecture terms (a prior bookmark design, a prior invalidation design, a replica-token-age metric, an object-store snapshot subsystem, a forward-compat window) that the reviewers treated as live requirements. Most UNCOVERED/WEAK findings were unusable; only ~3 of ~14 per reviewer were genuine after manual filtering. Mitigation for next run: (a) strip historical context from the spec before review, OR (b) pass the test plan content inline in the prompt rather than via path lookup, OR (c) explicitly instruct Codex "the Rewrite-history section is NOT a list of current requirements; only the live sections are."

- **[single happy-path test org masks fallback bugs]** — A send-to-channel feature shipped broken because all E2E ran against the one test org that happened to have the primary channel row, so the fallback path never fired. ALWAYS require ≥1 test org variant per fallback branch. Enforce via the surface-extractor's tenant-variance section.
- **[smoke-level Deploy Validation]** — 200-OK health probes passed while concurrent OAuth races, IAM propagation lag, and real object-storage signed-URL paths all failed in prod. The deploy tier must be chaos-level (IAM wait, real storage, concurrent writers, env-drop) not smoke.
- **["52/69 PASS" false confidence]** — a sandbox-lifecycle plan was structurally green, 16 prod bugs shipped. Coverage count without adversarial depth = false confidence. The Blast Radius Checklist is the gate against this.
- **["derive from spec, not imagination"]** — Prevented probing beyond spec text; specs don't enumerate failure modes. Changed to "derive primarily from spec, but go beyond with adversarial analysis."
- **[JSON templates in Codex prompts]** — Codex echoes them back as fake findings. Use prose structure + minimum-count requirements, not templates/markers.

## Patterns

- **Plumbing trace rule:** for every callback (onSuccess, completion callback, onRunTerminal), enumerate every terminal path separately. Unit test validates wiring; integration test validates firing in each path.
- **Stub detection rule:** for every `NotImplementedError` / `TODO` / placeholder return, the plan must include a test that fails if the stub is still present at ship.
- **Runtime-parity surface:** storage backend, IAM model, DB connection pooling, dependency version features — all need deploy-tier probes that exercise the real prod path, not local FileSystem/mocks.
- **Two-phase writes:** for every flag that drives async behavior, the spec must declare write-order (flag BEFORE invoke or after) and the test must verify.
- **Context bloat reduces success:** an ETH Zurich AGENTS.md study found auto-generated context files reduced success ~3% and inflated cost 20%+. Keep per-project config thin; let the LLM re-derive what's cheap to derive.

## What Failed (continued)

- **[`final-state-only` assertions hide pre-items pricing bugs]** — A follow-up rev asserted `Order.price == expected` but didn't verify the OLD signal path was quiet. A later refactor could reintroduce the double-call bug and the final price would still match (because the helper overwrites). Fix: spy-on-call-site asserting `order.order_items.count() > 0` at every helper invocation AND that the retired code path (signal) never fires pricing. Generalize: when centralizing a behavior, test the old path's silence, not just the new path's correctness.
- **[`get_or_create` seed ledgers can delete reused prod rows on teardown]** — A seed-run design recorded every PK in `created_pks`, but `get_or_create` returns existing rows unchanged. If a seed reused an existing prod User (matching email), teardown would delete that prod row. Always distinguish `created_pks` (where `created=True` returned) from `reused_pks`. Cite: Codex Adv CRITICAL.
- **[`ContextVar` guards don't cover `transaction.on_commit`]** — Django's `on_commit` callbacks fire AFTER the atomic block exits, by which time the `contextvars.ContextVar` is reset. Seed-time notification suppression that only checks `is_seeding()` at send-time misses on_commit-queued sends. Tests must capture on_commit callbacks, commit the transaction, then execute the callbacks and assert mocks still show zero calls.
- **[Raw-phone SMS sends bypass user-based demo guards]** — OTP / phone verification paths send SMS to a raw phone number BEFORE a User row exists. A `_is_demo_user(user)` check can't identify the recipient. Every SMS entrypoint must either (a) check `is_seeding()` as the primary guard, or (b) add a phone-number-prefix allowlist. Cite: Codex Adv HIGH.
- **[MX-null domain check insufficient for demo email safety]** — A demo domain with no MX can still route via A/AAAA fallback if SMTP clients are misconfigured. Full verification requires: authoritative NS/SOA/registrar ownership matches the target org; no MX; no A/AAAA; result recorded per-batch with fresh TTL so `--execute` refuses stale verification.

## Open Questions

- How to catch observability / log-noise gaps in test plans? They mostly surface post-deploy at production scale (a real incident: 497/500 ERROR slots were bot traffic for 7 days). Possible probe: require an ERROR-budget assertion per new log line, but unclear how to measure baseline in the test environment.
- When does a Chain test become overkill vs "just do E2E"? Heuristic unclear — currently "chain if multi-layer SDK→wrapper→adapter, else E2E."
- Contract testing for mobile decoders without running the mobile app — snapshot serialization shapes, but unclear how many fixtures cover the real decoder surface.

- **[scope-token primitive] Coverage-verifier "FAIL_POLICY direction" finding is a class to hunt on every security-flag spec.** A kill-switch defaults to OFF (`fail_safe`→`'0'`) which is correct for availability flags but BACKWARDS for a *security enforcement* flag — a transient config-fetch blip silently disables enforcement (getFlag returns the default BEFORE the fail-stale cached-config branch). Worse, a fail-closed companion (`enforcement_epoch`) and a fail-safe flag have OPPOSITE failure directions → an inconsistent per-request decision. Pattern: for any spec adding an enforcement/security config flag, the test plan MUST assert the blip direction (require fail_stale or key off the fail-closed value), and test the composite decision when the flag and its companion disagree under a failed fetch.
- **[scope-token] Token-refresh paths launder time-based fences.** A `refreshServiceJwt` re-mints with a fresh `iat`; a token minted before an `iat>=epoch` enforcement cutover acquires a post-epoch `iat` on cold-start refresh and sails past the fence. Mint-time tests don't catch it — needs a mint→refresh→enforce CHAIN test parameterized over the relevant variants. Pattern: whenever a spec adds an `iat`/version fence AND a token-refresh/rotation path exists, add a chain test proving refresh preserves the original `iat`/version, not just the scope claims.
- **[scope-token] CI-tripwire grep tests have three false-green vectors that must themselves be tested.** A grep asserting "minting only in approved factories" misses: variable-args (`generateServiceJwt(opts)`), key aliases/whitespace (`run_id` vs `runId`), spread shorthand (`...(x.runId?{runId}:{})`), and — meta — a path-typo'd matcher that 0-matches and reports green (the infra-synth false-skip class). Pattern: every grep/static-guard test needs a positive+negative FIXTURE (a known-bad sample that MUST trip it) so a broken matcher fails loud.
- **[scope-token] Parity ≠ correctness.** An allow/deny parity matrix across 3 enforcement surfaces passes even if all three share the same bug (null→allow). The parity test must be built on the corrected precondition (null→DENY) as an explicit row, not a sibling test that can pass while parity passes on the wrong value. Pattern: pair every "surfaces agree" parity assertion with a "they agree on the SECURE value" assertion.
- **[scope-token] Scoped-token fixture-integrity gate.** When the only CLI/dev JWT mints `'*'`/no-agent_id, every scoped test that reaches for the habitual in-repo mint helper silently tests the UNSCOPED path → "scoped read denied" passes for the wrong reason (the `'*'` token also can't reach an un-instrumented path that throws). Pattern: gate scoped tests on a shared helper that asserts non-`'*'`+agent_id+subject_scope before any test body, and grep-guard that scoped rows import only from it. Same root as the handler-async "CLI test JWT scope mismatch" class.
- **[scope-token] Infra reality audit's cross-worktree scan caught a live collision the main view couldn't see** (a sibling worktree editing the exact minting+flag surfaces — feature-flags / automation-executor / op-worker). Confirms the Step-1b audit's worktree scan is load-bearing for any spec touching shared platform files; it turned into a Blocking Dependency (merge coordination), not a test row.
- **[scope-token] Analysis agents found the spec's OWN code anchors partially false** (a builder returned unquoted refs + JSON-fallthrough-on-absence, not the spec's claimed quote+bind+1=0; `aud` was only enforced for sandbox-typed subjects). Pattern: surface/adversarial/edge-case agents reading the actual builder beat the spec-review's file:line citations because spec-review confirmed the anchor EXISTS, not that it BEHAVES as the spec claims. For predicate-injection/SQL specs, the test-plan analysis must read the reused builder's body, not just confirm its line number.
