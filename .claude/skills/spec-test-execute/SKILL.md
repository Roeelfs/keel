# Spec Test Execute — Test Plan Runner

Executes a test plan file generated by `/spec-test-plan`. Runs tiers sequentially (unit → integration → deploy → E2E), dispatches diagnosis agents for failures, escalates to Codex when stuck.

**Trigger:** "execute test plan", "run test plan", "run the tests"

**Prompt templates:** `prompts/infra-reality-auditor.md`, `prompts/failure-diagnostician.md`, `prompts/codex-rescue-stuck.md`

## Severity Taxonomy (shared across spec-review, spec-test-plan, spec-test-execute)

When executing a test plan, severity on each row is inherited from the plan. When surfacing new findings in the execution report (Step 6), use this scale.

| Unified | Codex (critical/high/medium/low) | Claude (CRITICAL/MAJOR/MINOR) | What it means |
|---|---|---|---|
| **CRITICAL** | critical | CRITICAL | Security, data loss, compliance, or rollback safety. Block ship. |
| **MAJOR** | high, medium (when user-facing) | MAJOR | Correctness, availability, contract break. Fix before release. |
| **MINOR** | low, medium (when internal-only) | MINOR | Performance, cosmetic, edge case. Can ship with a tracked follow-up. |

Failure-diagnostician and Codex rescue outputs normalize to this scale before being written back to the plan.

## Strict Rules (non-negotiable)

1. **Evidence before marking.** NEVER mark PASS without running the test and seeing output.
2. **Plan file is the ledger.** Update status after EACH test, not in batch.
3. **No silent skips.** Every SKIP/BLOCKED gets a specific reason + fallback attempt.
4. **Fix forward.** Diagnose root cause. Don't delete/weaken tests to make them pass.
5. **One tier at a time.** Don't start Tier 2 until Tier 1 is resolved.
6. **Max 5 cycles.** After 5 passes, remaining FAILs become BLOCKED with diagnostics.
7. **3-strike recurring failure.** Same error 3x → auto-BLOCKED, escalate to Codex rescue.
8. **Background for heavy ops.** Full suite runs and installs run in background.
9. **Fallback before SKIP.** Try alternative verification before accepting a SKIP.
10. **Assertion strength audit.** A PASS with weak assertions becomes GAP.

## Status Markers

| Marker | Meaning |
|--------|---------|
| `PASS` | Ran, all assertions passed |
| `FAIL` | Ran, assertions failed |
| `FIXED` | Previously FAIL, now passes after fix |
| `SKIP` | Cannot run, reason documented, fallback attempted |
| `BLOCKED` | Requires human action or 3x recurring failure |
| `GAP` | Test insufficient, needs strengthening |

## Flow

### Step 1: Load Plan, Test Config & Initialize State

Find the test plan:
```bash
ls -lt docs/specs/*-test-plan.md | head -5
```

Read the plan + referenced spec. Parse: total tests per tier, existing status markers (for resume).

**Also read `testing/config.md`** if it exists in the project root. This provides project-specific E2E patterns — staging URLs, test runners, test accounts, platform commands, known limitations. If the file doesn't exist, flag it:

> **No test config found.** Will create `testing/config.md` in Step 5.5 with any provisioning commands, timeout learnings, and infrastructure knowledge discovered during testing.

**Also read `testing/flows.json`** if it exists in the project root. This provides accumulated E2E flow knowledge — step-level operational notes, environment requirements, and verified gotchas from prior test cycles.

When flows.json exists, extract knowledge for the current spec's scope:
1. Scan flow names and descriptions for keywords matching the spec's features
2. For each matched flow, read step `notes` (protocol details, format quirks, timing), step `requires` (env vars), and flow `requires` (cross-cutting env vars)
3. Read flow `gotchas` — **treat each gotcha as blocking operational knowledge, not optional commentary.** A gotcha that says "wait for ai_message, not step_change" is a protocol instruction that prevents test failure.
4. Check all `requires` entries (flow + step level) and verify the env vars are set. Flag missing ones before starting tests.
5. Pass this operational context to the agent so E2E scripts use correct protocols, timeouts, and env var checks from the start.

If `testing/flows.json` doesn't exist, note: "No flow registry found — will bootstrap in Step 5.5 with any discoveries."

Check for resume state at `/tmp/.spec-test-execute-state.json`. If exists, resume from recorded phase.

If fresh, create state file:
```json
{
  "planPath": "<path>",
  "specPath": "<path>",
  "phase": "prerequisites",
  "cycle": 1,
  "maxCycles": 5,
  "failureHistory": {},
  "tierStatus": { "unit": "pending", "integration": "pending", "chain": "pending", "deploy": "pending", "e2e": "pending", "flowSync": "pending" }
}
```

Report: N total tests, N previously executed, N remaining, cycle N/5.

### Step 1b: Infra Reality Audit (MUST run before the prerequisite gate)

**Problem it solves:** Parallel worktrees drift. A session in `.worktrees/<feature>` misses tests, fixtures, or flow updates that sibling worktrees or `main` already have. Executing without this audit leads to duplicate test authoring and clobbering of in-flight work.

**Dispatch one `Explore` agent (sonnet, ~5 min)** using `prompts/infra-reality-auditor.md`. Pass:
- Plan path
- Spec path
- Project root
- Surface-area keywords extracted from the plan's tier rows
- `ls <project_root>/.worktrees/ 2>/dev/null` output

**Gating rule:** If the audit reports CROSS-WORKTREE in-flight test work relevant to this plan, HALT and ask the user whether to wait / rebase / coordinate before executing. Never silently proceed past this finding — it's the #1 way parallel work collides.

Feed the audit into execution:
- ALREADY IMPLEMENTED rows → re-run existing tests, don't author duplicates
- FIXTURES/HELPERS → use them directly, import from existing paths
- GAPS → the only rows that justify new test files

**Default stance: re-run what exists. Only scaffold new test code for rows in GAPS.**

### Step 2: Prerequisite Gate (strict — halt if blocked)

- Verify blocking dependencies (from plan's `## Blocking Dependencies`)
- Verify environment: services running, env vars set, migrations applied, tooling installed
- Run setup commands from plan if needed
- **HALT if any blocking dependency unresolved.** Tell user exactly what to fix.

**IMPORTANT: Staging deployment is NEVER a blocking dependency.** If the plan lists staging deployment or "code must be deployed" as a blocking dependency, ignore it — the Staging Auto-Deploy Protocol (Step 3) handles deployment automatically. Only halt for things that truly require human action outside the agent's capabilities (e.g., manual credential provisioning in a third-party system, hardware setup, org-level access grants).

### Step 3: Execute Tiers (sequential)

Execute in order: **Unit → Integration → Chain → Deploy Validation → E2E**.

**Tier 2.25 (Chain Tests):** If the plan has a `## Tier 2.25: Chain Tests` section,
run these after Integration and before Deploy Validation. Chain tests exercise
multi-step sequences end-to-end (upload→serialize→render; checkout→execute→checkin)
in the test environment but WITHOUT launching a browser. They catch plumbing drops
that unit+integration miss (e.g., a completion callback wired into only some of the
completion paths) and contract mismatches between serializer output and consumer
expectations.

**Deploy Validation is chaos-level, not smoke-level:** The plan's Tier 2.5 probes
should exercise real prod paths (cloud-IAM credential signing, real object-store URL
signing, concurrent writers, env-drop fallbacks, connection-pool saturation). A shallow
"200 OK" health check is insufficient. If the plan only has shallow probes, mark them as
GAP and propose stronger ones via the diagnostician.

**Per-test execution:**
1. Write the test (following project conventions from plan)
2. Run it (capture full output)
3. Evaluate result → mark in plan file immediately
4. If FAIL → dispatch diagnosis (see Step 4)

**Per-tier completion:**
- Run full tier suite for regression check
- Report: N/N PASS, N FAIL, N SKIP, N BLOCKED
- **Gate (Unit & Integration only)**: Do not advance to the next tier if any FAIL remains in Unit or Integration.
- **NO gate on E2E.** Deploy Validation and E2E tests ALWAYS execute regardless of whether deployment is "ready." See E2E rules below.

**E2E execution rules — NEVER skip, NEVER gate on deployment:**

E2E tests are where the real bugs live. Skipping them because "staging isn't deployed yet" or "infrastructure isn't running" defeats the entire purpose of testing. The skill MUST always attempt to run E2E tests — including deploying to staging if needed.

1. **Read the project test config** (`testing/config.md`) for staging URLs, runners, test accounts, deploy method, and commands. If no config exists, use the test plan's prerequisites section.

2. **Check staging availability first:**
   ```bash
   # Use the health check from test config, e.g.:
   curl -sf <STAGING_URL>/api/health/ --max-time 5
   ```
   - If staging responds → proceed with E2E tests against staging
   - If staging is unreachable → **deploy automatically using the Staging Auto-Deploy Protocol below**

3. **Staging Auto-Deploy Protocol (when staging is unreachable or stale):**

   This is the CRITICAL section. When E2E tests need staging and it's not available, DO NOT stop and ask the user. Deploy it yourself.

   **a. Determine deploy method** from config.md's `## Staging Deployment` section:
   - If a `Deploy command` field exists (e.g., `<your deploy-to-staging command>` from `testing/config.md`) → run it directly
   - If deploy method is "push to staging branch" → execute:
     ```bash
     git push origin HEAD:staging
     ```
   - If neither is documented → try `git push origin HEAD:staging` as default (most projects auto-deploy on staging branch push)

   **b. Wait for CI deployment** (for push-based deploys):
   Read the `CI workflow` field from config.md to get the workflow filename. Then poll:
   ```bash
   # Get the latest workflow run triggered by the push
   gh run list --branch staging --workflow <WORKFLOW_FILE> --limit 1 --json status,conclusion,databaseId
   ```
   Poll every 30 seconds, max 10 minutes total. Check for `conclusion: "success"`.

   If the CI run fails, capture the failure:
   ```bash
   gh run view <RUN_ID> --log-failed 2>&1 | tail -50
   ```
   Log the CI failure in the execution report, then attempt to run what E2E tests are possible without staging.

   **c. Health check after deploy** — retry with backoff:
   ```bash
   # Retry 5 times, 15-second intervals
   for i in 1 2 3 4 5; do
     curl -sf <STAGING_URL>/api/health/ --max-time 10 && echo "HEALTHY" && break
     sleep 15
   done
   ```

   **d. If deploy succeeds** → run canary automation if the project has one (check `testing/config.md` for `## Canary Automation`). The canary verifies runtime integrity (e.g. function layers, imports) that health checks don't catch. If canary fails, check the runtime logs before proceeding — a silent runtime failure will cause all automation E2E tests to fail with misleading errors.

   **e. If deploy fails** → run what you CAN (API tests, unit-level UI tests, build output checks). Mark only truly impossible tests as BLOCKED with reason: `staging deploy failed — CI error: <summary>`.

   **NEVER stop execution to ask the user to deploy.** The whole point of this protocol is autonomous E2E execution.

4. **Run EVERY E2E test in the plan.** For each:
   - Check runner type (Playwright, Chrome Extension, iOS Simulator, Sandbox Agent, etc.)
   - Use the project test config's commands and accounts
   - Playwright: write test file, run with project config, screenshot on failure
   - Chrome Extension: use MCP tools (tabs_context → create → navigate → interact → assert)
   - Mobile: use simulator commands from test config
   - Sandbox Agent: use CLI commands from test config
   - Use `gif_creator` for visual evidence on complex flows

5. **If a test can't execute, apply Rule 9 (fallback verification) before accepting SKIP.** The test config's "Known Test Environment Limitations" section provides fallback strategies.

6. **Flag staging issues prominently in the execution report** — but never let them silently block the entire E2E tier. A test plan where E2E is all-SKIP is a failed test execution, not a successful one.

### Step 4: Failure Diagnosis (parallel agents)

When tests fail within a tier, dispatch **failure diagnostician agents** in parallel — one per failing test (see `prompts/failure-diagnostician.md`):

- **Type:** `general-purpose` | **Model:** `sonnet`
- **Input:** test ID, error output, spec section, file path
- **Output:** diagnosis category (test-wrong / implementation-wrong / infrastructure / spec-ambiguity) + proposed fix

Apply fixes from diagnosticians, re-run tests, mark FIXED or increment failure count.

**Escalation to Codex (see `prompts/codex-rescue-stuck.md`):**

When a test has failed 2+ times and the Claude diagnostician couldn't fix it, run Codex rescue via Bash (**NEVER use Skill tool** — `disable-model-invocation: true`).

Unlike `review` / `adversarial-review` (which feed Codex git diffs and are wrong for file reviews), **managed `task` dispatch is the correct tool here** — it takes a task prompt with write-mode sandbox access, which is exactly what fixing a failing test requires. The managed dispatcher pins repo context, passes prompts safely, enables network for setup by default, and has a watchdog.

**Use `run_in_background: true` on the Bash call** — Codex fixes can take 10+ minutes. Foreground Bash times out at 10 min max.

```bash
# Primary: managed dispatcher task (write-capable task delegation)
# MUST use run_in_background: true on the Bash tool
TASK_TAG="rescue-<TEST_ID>-$$"
cd <PROJECT_ROOT> && codex-dispatch task --network on --fast -- \
  "Fix the failing test <TEST_ID>. Error: <ERROR_OUTPUT>. Spec section: <SPEC_SECTION>. File: <FILE_PATH>. The Claude diagnostician tried: <DIAGNOSIS>. Fix the implementation or the test — do not weaken assertions." \
  > /tmp/codex-${TASK_TAG}.txt 2>&1
```

**Monitoring:** Liveness-based, no wall-clock deadline. Rescue runs routinely take 20-45+ min because write-capable fixes require: diagnose → patch → run tests → verify → iterate. Killing mid-iteration produces partial fixes that leave the tree in a worse state than before.

Checkpoint cadence: first at T+5 min, then every ~5 min via `ScheduleWakeup`. Carry previous output file size between checkpoints. At each:
1. **Status marker present?** `grep -q "CODEX_RESCUE_STATUS" /tmp/codex-${TASK_TAG}.txt` → parse the latest status block — `test-passed` means done, `blocked` means give up, `fix-applied`/`diagnosis` means keep waiting
2. **Process exited?** `pgrep -f "codex-dispatch task.*<TEST_ID>"` returns nothing → read output, dispatcher finished
3. **File growing?** Current size > previous → alive, keep waiting
4. **File idle ≥10 min with process alive?** → the dispatcher watchdog should kill and reconcile automatically; if not, kill the dispatcher and run the dispatcher's `reconcile` command

**Never SIGTERM on wall-clock.** A 40-minute rescue that fixes the test is a success. Only kill on sustained idle or on explicit `blocked` status.

**Fallback: raw `codex exec` with network-enabled sandbox**

If the managed dispatcher crashes or exit code ≠ 0 before Codex starts, fall back to raw `codex exec`, but with `danger-full-access` so Codex can modify files and fetch dependencies:

```bash
TASK_TAG="rescue-fallback-<TEST_ID>-$$"
cd <PROJECT_ROOT> && echo '' | codex exec --skip-git-repo-check \
  -m gpt-5.5 \
  --config model_reasoning_effort="high" \
  --config service_tier="fast" \
  --sandbox danger-full-access \
  --full-auto \
  "$(cat <<'PROMPT'
<task>
Fix failing test <TEST_ID>.
Error: <ERROR_OUTPUT>
Spec section: <SPEC_SECTION>
File: <FILE_PATH>
Claude diagnostician already tried: <DIAGNOSIS>

Fix the implementation or the test — do not weaken assertions, do not delete the test, do not mock the dependency that's the actual subject of the test.
</task>

<action_safety>
Only modify files directly related to this test failure.
Do not refactor surrounding code or fix unrelated issues.
Do not mass-rename or re-format.
</action_safety>

<verification_loop>
After applying a fix, run the test command and verify it passes.
If it still fails, investigate deeper — don't retry the same approach.
Emit progress using the findings contract below so the coordinator can track you.
</verification_loop>

<output_contract>
At each checkpoint (initial diagnosis, fix applied, test run result), emit:
===CODEX_RESCUE_STATUS===
{"phase": "diagnosis|fix-applied|test-passed|test-failed|blocked", "summary": "...", "files_modified": ["..."]}
===CODEX_RESCUE_END===

At the end of the session, emit one final status block with phase "test-passed" or "blocked".
</output_contract>
PROMPT
)" 2>&1 | tee /tmp/codex-${TASK_TAG}.txt
```

**Key differences from file reviews:**
- `--sandbox danger-full-access` — Codex needs to modify files and may need package/network setup to fix the test
- No wall-clock budget — same liveness-based monitoring as the primary dispatch above. Rescue fixes can legitimately take 30-60+ min for gnarly failures (flaky tests, deep stack issues, multi-file patches).
- Progress marker is `CODEX_RESCUE_STATUS` (not `CODEX_FINDINGS_BEGIN`) — rescue emits multiple status blocks as it works. Use the latest status block as the signal, plus the same liveness checks (process alive + file growth).

Read the output file when notified of completion. If Codex (either primary or fallback) still cannot fix → BLOCKED with both diagnoses for human review.

### Step 5: Loop Check (bounded, failure-aware)

After each pass:
1. Increment cycle in state file
2. Scan for non-terminal items (not PASS/FIXED/SKIP/BLOCKED)
3. Check cycle budget (max 5)
4. Check failure history (3x same error → auto-BLOCKED, escalate to Codex)
5. Process remaining: re-diagnose FAILs, execute unmarked, write GAP tests
6. Loop or proceed to Step 6

### Step 5.5: Knowledge Sync (Flows + Test Config)

After the test loop resolves and before writing the execution summary, sync accumulated knowledge to both the flow registry and the test config.

**If `testing/flows.json` doesn't exist:** Create it with `{"version": 1, "flows": {}}` using the Write tool.

**If `testing/flows.json` is malformed:** Warn the user and skip this step. Do not attempt to auto-fix.

**If `testing/flows.json` exists and is valid:** Read it, then perform the sync:

1. **Review E2E execution results.** For each tested behavior, decide:
   - Is this a reusable primitive (login, OTP, deploy) or a product-specific journey (e.g. a multi-step account-setup flow)?
   - Would another journey reuse this step? If yes → primitive. If no → journey step with inline `action`.

2. **Check existing flows.** For each flow you plan to create/update:
   - Read `flows.json` and check if the flow ID already exists
   - If exists: update `description`, `how_to_verify`, `steps`, and `updated_at`. Do NOT change `type`, `product`, or `created_at`.
   - If new: create the full flow object with all required fields. Use current UTC ISO 8601 timestamp.

3. **Record run results.** For each flow that was tested:
   - Prepend a run entry with: `spec_ref` (current spec name), `commit` (current HEAD), `status` (mapped: PASS/FIXED→pass, FAIL→fail, SKIP→skip, BLOCKED→blocked, GAP→partial), `runner`, `notes`, `run_at` (current UTC ISO 8601)
   - If the flow now has >20 runs, drop the oldest entries

4. **Capture gotchas.** For non-obvious findings discovered during testing:
   - Check existing gotchas on the flow — skip if already recorded with `still_relevant: true`
   - Only record findings an agent would benefit from knowing next time (CSRF domains, rate limits, encoding issues, env-specific config)
   - Do NOT record obvious outcomes ("page loads", "button works")

5. **Extract step notes.** For every E2E step executed, check: did the agent encounter something unexpected about the protocol, timing, or format? Compare what the test plan said vs what actually worked. Write operational details into the step's `notes` field. Use the knowledge ownership table: step-specific protocol goes in `notes`, not in gotchas or config.md.

6. **Extract step and flow requires.** For every env var or flag that was set mid-session to unblock a test, was missing and caused a failure, or was needed for a workaround — add to the appropriate `requires` array. Step-level if the requirement is specific to one step; flow-level if it applies across multiple steps.

7. **Migrate step-specific gotchas.** If a flow-level gotcha clearly maps to a single executed step (e.g., "EIN format must be XX-XXXXXXX" → business verification step), move it to that step's `notes` field and remove the gotcha entry. Only migrate with high confidence — ambiguous gotchas stay flow-level. This is a semantic change and goes through the approval gate.

8. **Ask: "What did I learn that would have saved me time?"** After extracting specific facts, reflect broadly: protocol discoveries, timing learnings, provisioning steps that worked, format quirks. Write anything new to the appropriate destination per the knowledge ownership table.

#### Approval Gate

Separate extraction into two tiers before writing:

**Auto-apply (no approval needed):** Run records, flow metadata updates (description, how_to_verify, steps). These are structural facts, not interpretations.

**Present for approval:** Step `notes`, step/flow `requires`, gotcha migrations, config.md section additions/updates. These are semantic knowledge where the LLM's interpretation could be wrong.

Present semantic changes as a grouped diff:

> **Proposed flows.json changes:**
> - `flow-id.steps[N].notes` → "new operational note..."
> - `flow-id.steps[N].requires` → `["ENV_VAR=value"]`
> - `flow-id.requires` → `["CROSS_CUTTING_VAR=true"]`
> - Move gotcha "..." → step[N] notes
>
> **Proposed config.md changes:**
> - Add/update `## Test Data Provisioning` (N entries)
> - Add/update `## Default Timeouts` (N entries)
>
> Apply? [yes / edit / skip]

**Re-read before write:** After the user approves, re-read `testing/flows.json` and `testing/config.md` before writing. If the file changed since the diff was generated, regenerate the diff against the new state and re-present. This preserves the fresh-state write invariant from the flow registry spec.

9. **Compose journeys.** For multi-step flows:
   - Add `ref` steps pointing to primitives (verify the target exists and is active)
   - Add inline `action` steps for journey-specific behavior
   - Check for cycles: verify the target doesn't transitively reference this flow

10. **Write and verify.** Edit `testing/flows.json` using the Edit tool. Then re-read the file and verify:
   - Valid JSON
   - All invariants hold (step XOR, ref targets exist, no cycles, runs ordered correctly)
   - If verification fails, warn and attempt to fix. If unfixable, revert the edit.

11. **Stage for commit.** Run `git add testing/flows.json` so the registry is included in the spec-test-execute commit.

#### Part B: Test Config Sync

Review what was encountered during test execution vs what `testing/config.md` currently documents. Update the config with any newly discovered infrastructure knowledge.

**What to look for:**

1. **Env vars / feature flags** — did any test require an env var or flag not listed in `testing/config.md`? Add it with the value or instructions for setting it.

2. **Service dependencies** — did any test depend on an external service (AI API, payment gateway, VPC connector, scheduler) not listed? Add it with availability notes and fallback behavior.

3. **Rate limits** — did any test hit a rate limit not documented? Add the endpoint, threshold, and workaround.

4. **Test accounts** — did any test need an account state (pending_review, rejected, specific role) that doesn't exist in the test accounts section? Add the needed account or flag that the seed script needs updating.

5. **Known limitations** — did any test hit a platform limitation (headless browser, simulator, API unavailability) not listed? Add it with the fallback strategy used.

6. **Runner config** — did any test require runner-specific setup (image fixtures, WS client config, parallel worker limits) not documented? Add it to the relevant runner section.

7. **Test Data Provisioning** — did the agent have to provision test accounts, change account status, run seed scripts, or use admin UIs? Capture the commands that *worked* into a `## Test Data Provisioning` section. Include workarounds and known dead-ends.

8. **Default Timeouts** — did the agent change timeouts during execution (started at 30s, bumped to 120s)? Record final working values in a `## Default Timeouts` section, organized by step type (GPT-powered, file upload, simple form, full flow).

**How to update:** Read `testing/config.md`, identify the correct section for each new item, and edit it directly using the Edit tool. Append new items to existing sections — don't reorganize or rewrite existing content. Re-read after editing to verify valid markdown.

**If `testing/config.md` doesn't exist:** Create it with any provisioning commands, timeout learnings, and infrastructure knowledge discovered during this session. This bootstraps the config file so future sessions start with accumulated knowledge. Present the proposed content for approval before creating.

**Stage for commit.** Run `git add testing/config.md testing/flows.json` so both files are included in the spec-test-execute commit.

- **State file** (`/tmp/.spec-test-execute-state.json`): phase, cycle, failure history
- **Plan file status markers**: PASS/FIXED skip, FAIL re-run, unmarked execute fresh

State file takes precedence for phase/cycle. Plan file for individual test status.

## When NOT to Use

- No test plan file exists — run `/spec-test-plan` first
- Implementation hasn't started — plan is for post-implementation verification
- Plan already fully EXECUTED with all PASS — nothing to do
- Single test to run — just run it directly
